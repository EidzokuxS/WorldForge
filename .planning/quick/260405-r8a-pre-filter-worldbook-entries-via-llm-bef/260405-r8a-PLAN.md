Add a new async function `filterRelevantEntries()` in composition.ts that filters supplementary source entries via Judge LLM call.

**Function signature:**
```typescript
async function filterRelevantEntries(
  premise: string,
  primarySourceName: string,
  supplementaryRecords: StoredWorldbookLibraryRecord[],
): Promise<Map<string, Set<string>>>
```
Returns a Map from record.id to Set of entry names that the Judge deemed relevant.

**Implementation steps:**

1. Build a concise index of ALL supplementary entries across all supplementary records:
   - Format: `"[SourceName] EntryName (type): one-line summary"` — one line per entry
   - Keep it compact — the Judge only needs enough to decide relevance

2. Define a Zod schema for the Judge response:
   ```typescript
   const filterSchema = z.object({
     relevantEntries: z.array(z.string()).describe("Entry names from the supplementary sources that are relevant to this campaign premise"),
     reasoning: z.string().describe("Brief explanation of selection criteria"),
   });
   ```

3. Call `safeGenerateObject` with Judge role (temperature: 0, retries: 1):
   - Prompt framing (POSITIVE only, per user feedback):
     - State the premise and primary source name
     - Present the supplementary entry index
     - Ask: "Select supplementary entries that would enrich this campaign. Focus on entries that complement the primary world's themes, setting, or conflicts. Include entries that the premise explicitly references or implies."
     - Tell the Judge to return the EXACT entry names from the index

4. Parse the result: build `Map<recordId, Set<entryName>>` by matching returned names against each supplementary record's entries (case-insensitive matching, same fuzzy approach as detectPrimarySource)

5. On failure (catch block): log warning, return empty Map (which signals the caller to include all entries — graceful fallback)

**Wire into composeWorldbookLibraryRecords:**

After `detectPrimarySource` runs and before building `sourceGroups`, when `primarySourceName !== null` (meaning there IS a primary/supplementary split):

1. Collect supplementary records: `sortedRecords.filter(r => r.displayName !== primarySourceName)`
2. Call `filterRelevantEntries(premise, primarySourceName, supplementaryRecords)`
3. If the returned Map is empty (LLM failure or no filtering needed), keep all entries (current behavior)
4. If the Map has entries, filter each supplementary record's entries before passing to `extractSourceData`:
   ```typescript
   const allowedNames = relevanceMap.get(record.id);
   const filteredEntries = allowedNames
     ? record.entries.filter(e => allowedNames.has(e.name.toLowerCase().trim()))
     : record.entries; // fallback: include all
   ```
5. Use `filteredEntries` instead of `record.entries` when building `sourceGroups` for supplementary sources
6. ALSO filter the entries that go into the merged `grouped` Map and ultimately `ipContext.keyFacts` — supplementary entries not in the relevance set should be excluded from BOTH flat keyFacts and sourceGroups keyFacts

**Important:** The merged `grouped` Map built at line ~184 currently iterates ALL entries from ALL records. Restructure so that supplementary record entries are pre-filtered BEFORE the grouping loop. One approach: create a `getFilteredEntries(record)` helper that returns filtered entries for supplementary records and all entries for primary records, then use it in both the grouping loop and the sourceGroups building.

Log the filter results: `log.info("Filtered supplementary entries: kept X of Y from source Z")`
