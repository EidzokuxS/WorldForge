# WorldForge Visual Target Contract

## Purpose

This is the visual north star for the play surface. It complements `worldforge-screen-flow-contract.md`.

The screen-flow contract answers "what screens and transitions exist." This document answers "what should it feel like when the player sees it."

## Hard Correction

The first Open Design prototype is structurally useful but visually wrong. It reads like a newspaper/editorial reader:

- cream page
- serif text columns
- small top tabs
- flat pills
- little sense of physical place
- no strong game frame
- no visual scene as the first-viewport signal

That direction is rejected for live play.

## Target Feeling

WorldForge should feel like:

- a solo RPG table becoming a visual novel scene,
- a playable scene with readable prose,
- a GM screen hidden behind the curtains,
- a living place the player can inhabit, not a document they are reviewing.

The player should feel:

1. "I am in this location."
2. "I can see who/what is actually present."
3. "The world is breathing even if I only continue."
4. "The system is deep, but it is not screaming debug data at me."

## Visual References To Chase

Use these as functional references, not skins to copy:

- Visual novel scene composition: background first, text box second, actors/portraits as presence.
- Marinara Engine game mode: scene image/weather/time surface + bottom narration box + lightweight widgets.
- Solo RPG journal/table feel only in review/setup surfaces, not the main play screen.
- Cinematic RPG dialogue UI: location and party/actor presence are visible, mechanics are tucked away.

## Marinara Interaction Reference

The Marinara reference is not a UI kit to clone. Its base UI is only a reference-quality midpoint: useful in flow, not sufficient in polish. WorldForge should take the parts that create immersion and improve the product language around them.

The useful lesson is that text becomes a staged interaction layer.

Observed target behaviors to preserve in WorldForge design:

- **Full-scene background** selected/generated by the GM layer. The location is visible before reading the paragraph.
- **Bottom VN narration box** with a title like `Narration`, current beat text, and controls.
- **Next button** reveals the next text segment instead of dumping the whole turn as one wall.
- **Autoplay** advances beats like a visual novel.
- **Log** opens the full prior text history without making the default screen a scrollback log.
- **Inventory** opens as an inventory, with item affordances, not as JSON/list admin.
- **Party chips/portraits** sit over the scene; clicking a party member opens their character card.
- **Floating HUD widgets** show map, sanity/clarity/bonds/clues/etc. as game instruments.
- **Journal/records** track met people, unknown people, inventory, clues, and GM/session notes.
- **Address modes** let the player write to the GM, to party members, or perform a scene action.
- **Choices / CYOA buttons** can appear when the GM offers them.
- **Quick-time style prompts** can appear when timing/pressure matters.
- **Dice/probability moments** are presented as game beats, not raw debug panels.
- **Text effects** can color, distort, glitch, emphasize, or animate words when fiction warrants it.
- **Screen effects** include fade to black for sleep/time skip, light flashes, tint shifts, weather overlays, and other scene transitions.
- **Sound/image controls** can live as small top controls, not as primary content.

WorldForge should copy these functions while keeping its stronger backend authority. Presentation effects are display events layered on top of settled world state; they do not become the source of truth.

Selection rule:

- Keep a reference element only if it answers a WorldForge player question or supports a real WorldForge mechanic/data surface.
- Improve the visual execution instead of copying Marinara's exact controls, spacing, panels, or widget chrome.
- Do not add widgets just because Marinara has them. Every widget needs a job: map, party, inventory, log, journal, clues, bonds/relations, status, GM notes, or settings.

## Visual References To Avoid

Avoid:

- newspaper/editorial reader
- Notion/document page
- SaaS dashboard
- debug cockpit
- generic purple sci-fi dashboard
- card pile inside card pile
- white/cream page with tiny navigation and long prose column
- visual hierarchy where raw text dominates everything

## Live Play Layout Target

Default `/game` should be scene-first:

```text
+------------------------------------------------------------+
| thin game HUD: location / time / weather / small controls  |
+------------------------------------------------------------+
|                                                            |
|       atmospheric scene backdrop / illustrated layer        |
|                                                            |
|          visible actor silhouettes / portraits / chips      |
|                                                            |
|                                                            |
|  +------------------------------------------------------+  |
|  | latest narration beat / dialogue / consequence       |  |
|  | freeform input + Continue + compact action chips     |  |
|  +------------------------------------------------------+  |
+------------------------------------------------------------+
```

Key requirements:

- The first visual read is a place, not a text page.
- The scene backdrop may be generated image, hand-picked placeholder, or stylized visual layer, but there must be a visual environment.
- Narration lives in a bottom or lower-center VN/game text box/dock, not as a newspaper column.
- The latest turn is segmented into beats with `Next`, `Auto`, and `Log` controls.
- Actor presence is spatial/visual: foreground chips, silhouettes, portraits, or scene markers.
- Party/nearby visible actors can be portrait chips over the scene; clicking opens character cards.
- Same-area/off-screen actors are not shown as present actors; they belong in World drawer.
- Oracle/debug/lore are overlay drawers, never default columns.
- Input belongs to the scene surface; it should feel like "act from here."
- Inventory, map, clues, bonds, sanity/clarity-like meters, and notes are floating game widgets or drawers, not permanent admin columns.
- Choices/QTE/dice prompts may appear above or inside the narration box as temporary game beats.
- Text and scene effects are valid presentation events: glitch text, colored words, fade-to-black, flash, weather/tint overlays, and speaker/thought callouts.

## Setup And Review Layout Target

Setup/review can be calmer and more utilitarian, but still not plain admin:

- Campaign Library: game launcher / save select, not table admin.
- Campaign Seed: guided authoring desk with source cards and premise composer.
- Worldgen Running: ritual/progress board with stages, not log spam.
- World Review: campaign prep table with map/cast/lore as reviewable artifacts.
- Character Handoff: character dossier + entry scene preview.

These screens may use denser panels, but they should still feel like preparing a game, not editing a database.

## Whole Product Surface Map

The design system must cover more than `/game`.

| Surface | Desired feel | Reference function |
|---|---|---|
| Main menu / campaign library | game launcher / save select | continue, create, delete/load without admin clutter |
| Campaign seed | guided world authoring desk | premise/source selection, imports, model/research readiness |
| DNA setup | tone/constraint shaping board | optional knobs that feel creative, not form-heavy |
| Worldgen progress | ritual/progress board | stages, elapsed time, recoverable background job, compact debug |
| World review | campaign prep table | map, locations, factions, lore, cast placement, targeted regenerate |
| Character add/import | character casting desk | imported/generated/manual characters with evidence and editable cards |
| Character card pre-play | dossier + fiction preview | identity, personality, powers, loadout, starting placement |
| Character card in-play | scene-aware character sheet | visible facts, relationship, status, inventory, advanced record hidden |
| Live play | VN/RPG scene surface | background, beat cadence, widgets, presentation events |
| Settings | equipment room / technical setup | providers, roles, gameplay toggles, debug hidden from core play |

Design work must explicitly name which surface it is improving. A visual decision that works for live play may be wrong for world review, and a review-table pattern may be wrong for live play.

## 2026-05-02 Reality Alignment Corrections

The current visual direction is close enough to keep, but it must be brought back to real WorldForge data and real user flows before Phase 77 implementation. The prototype must stop inventing product concepts just because they look good on a screen.

### Live Play

- The scene stage needs useful non-JSON game information in the empty side space: player mini-card/status, current location/map hint, visible/interactable actors, inventory/journal affordances, or scene clocks. The right floating rail must feel attached to the scene, not orphaned in black space.
- The VN dock needs a centered, comfortable reading lane. The latest text, `Next`/`Auto`/`Log`, `Continue`, and freeform input should live in one local interaction zone so the player is not scanning left for prose and far right for the next button.
- Remove player-facing `chunk 1/3` style counters. Beat segmentation is implementation state, not a user goal.
- Remove `Speak` as a required button. Player input is always freeform scene text; the player should see the last displayed beat and the draft input at the same time.
- Transient scene messages, whispers, and floating dialogue must persist until the player advances with `Next`; no auto-disappearing text for information the player may need to answer.

### Home And Campaign Library

- `Play`, Home `Continue Session`, and a campaign card click cannot all appear as unrelated controls that do the same thing. The design must name one primary resume action and make campaign cards clearly browse/select/open behavior.
- Do not use invented lifecycle statuses such as `Active`, `Paused`, `Retired`, `Bleeding`, or fake `Playing` tags unless the backend actually has those states. Use real metadata: campaign title, last turn/save, current location/character, creation/generation readiness.
- Rename or remove `Worlds in Progress` unless it has a precise product meaning. Current campaigns are simply campaigns/worlds; incomplete generation is a separate state.

### World Generation And Review

- Generation has ten post-DNA steps in the current product flow; prototypes must not compress it to a fake five-stage ritual.
- World DNA fields are longer than tagline cards. Each DNA card needs enough room for one to five sentences, a symmetric grid, and clear reroll controls per field plus reroll-all.
- World Review must expose the real review surfaces: Premise, Locations, Factions, NPCs, Lore. It must support item-level regenerate/reroll/edit, not only a pretty summary.
- Character/NPC review cards need a clear drill-in state for full details: persona, personality, voice, goals, power stats, hax, vulnerabilities, location/scene/faction placement, and advanced record.
- Reduce theatrical labels. `Cast of named souls` should become plain product language such as `Characters`, `NPCs`, or `Cast`.

### Character Import And Player Character

- There is no standalone character-import page in the current product. Imported/generated NPC creation belongs inside the generated campaign review, specifically the NPC tab before campaign start.
- NPC creation modes are: describe, AI generate, research archetype, import V2 card; import mode is native resident vs outsider.
- Player character creation is a separate flow after the world is saved and the user chooses to continue to character creation.
- Do not show fake import roles such as `Flashback voice` or `Ghost` unless a future phase explicitly designs and implements them.

### Copy Rule

WorldForge can feel literary without naming every section like a prophecy. Use dramatic typography for the world and scene, but keep controls and review labels human, direct, and aligned with actual data.

## Color And Material

The rejected prototype overused warm cream and serif text until it felt like paper.

Better direction:

- Base: deep charcoal, rainy midnight, or muted cinematic neutral for live play.
- Accent: controlled amber/gold for warmth, plus cyan/blue-green for Shibuya rain/neon and inspect signals.
- Surface: translucent dark glass, lacquered panels, or smoky paper panels over a scene, not full-page parchment.
- Setup surfaces may be lighter, but live play should not be a white/cream document.

Avoid one-note beige/cream/sand dominance.

## Typography

Readable prose matters, but typography must not turn the app into a magazine.

Rules:

- UI text: compact, modern, game-like sans.
- Narration: readable literary font is okay, but inside a game text box.
- Avoid huge serif body columns.
- Avoid newspaper-style turn dividers as primary structure.
- Dialogue should have speaker/name treatment if a character is speaking.

## Motion And Feedback

Use restrained game feedback:

- scene fade / rain shimmer / time-of-day tint
- drawer slide over dimmed scene
- actor focus pulse when a character enters or speaks
- latest beat reveal / skip / history
- turn processing as compact "GM thinking / Oracle / settling" status
- `Next` advances beat segments; `Auto` advances with a readable delay.
- sleep/time-skip can fade to black, then fade into the new scene state.
- supernatural or high-pressure beats can briefly distort text or flash/tint the scene.

Do not animate everything. Motion should clarify state transitions.

## Visual Asset Rule

Live play prototypes must include a scene visual asset or a strong stylized scene layer.

Acceptable for prototype:

- generated bitmap scene background,
- searched/reference background if licensing is only for local mockup,
- CSS/HTML illustrated scene only if it convincingly reads as a place,
- blurred/tinted background plus explicit foreground scene objects.

Not acceptable:

- no visual scene,
- blank cream/white background,
- only text and pills.

## Required Prototype Checks

Before accepting a prototype:

1. Five-second screenshot test: does it look like a game/VN, not a webpage article?
2. Scene test: can the user identify the current location visually before reading all text?
3. Actor test: are visible/interactable actors visually distinct from off-screen/same-area actors?
4. Input test: does the input feel attached to the current scene, not pasted under a document?
5. Debug test: can raw system info be opened, and is it hidden by default?
6. Mobile test: does the scene/text/input stack preserve the game feeling?
7. VN cadence test: can a user read one beat, press `Next`, open `Log`, and return without losing input draft or scene state?
8. Presentation-event test: can a turn show at least one non-prose cue (speaker line, text effect, fade, flash, QTE, or dice beat) without becoming debug noise?

## Prompt Addition For Future Open Design Runs

Any future Open Design prompt must include:

> The prior warm parchment/newspaper/editorial-reader look is rejected. Do not create a cream document page with prose columns. The live play screen must look like a playable VN/RPG scene: scene backdrop or strong visual environment, lower narration/input box, actor presence, overlay drawers, cinematic mood/time/weather. Setup screens can be calmer but must feel like game preparation, not database admin.

Also include:

> Use Marinara-style presentation functions: full-scene background, bottom VN text box, Next/Auto/Log cadence, floating game widgets, party portraits, inventory/journal drawers, address modes for GM/party/action, CYOA/QTE/dice presentation beats, and selective text/screen effects. Preserve WorldForge backend authority: these are display events over settled state, not authoritative state by themselves.
