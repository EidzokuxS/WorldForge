# Phase 19.1: Remove Gameplay Fallbacks — Context

**Gathered:** 2026-03-20
**Status:** Ready for planning

<domain>
## Phase Boundary

Remove all gameplay-degrading fallbacks from core systems. Replace silent degradation with retry logic using the configured Fallback role (alternative model). This is a CODE CHANGE phase, not a testing phase.

### What to fix:

1. **Oracle coin flip** (`backend/src/engine/oracle.ts:129-139`): When Judge LLM fails, currently returns `chance: 50` (coin flip). Must retry using Fallback role model, then throw if both fail.

2. **Storyteller empty narrative** (`backend/src/routes/chat.ts` + `backend/src/ai/storyteller.ts`): When Storyteller LLM fails/rate-limits, currently returns 0-character narrative. Must retry using Fallback role model, then throw if both fail.

3. **Lore extraction empty** (`backend/src/worldgen/scaffold-generator.ts:330-345`): When lore extraction fails, currently returns `loreCards: []`. Must retry with smaller batch size or Fallback role, then return partial results if still failing.

4. **Premise retrieval empty** (`backend/src/routes/chat.ts:230`): When premise retrieval fails, currently returns `""`. Premise is essential for gameplay — must throw.

### What NOT to touch:

- `safeGenerateObject` two-tier (format recovery — this is fine)
- Settings normalization (infrastructure)
- NPC/faction tick error capture (independent subsystems, non-blocking is correct)
- Image/portrait fire-and-forget (optional features)
- Provider resolution fallbacks (infrastructure)
- Schema coercion (type recovery)

</domain>

<decisions>
## Implementation Decisions

### Claude's Discretion
- The Fallback role is already configured in Settings (settings/manager.ts). Use `resolveRoleModel('fallback', settings)` to get the fallback provider/model.
- Retry pattern: try primary role → on failure, try Fallback role → on both fail, throw AppError
- For lore extraction: try full batch → on fail, try in 2 smaller batches → on fail, try Fallback role → on both fail, throw
- Add `withFallbackRetry(primaryFn, fallbackFn)` utility if needed, or inline the retry logic
- Keep error logging (log.warn on retry, log.error on final fail)
- Quality threshold: after fix, Oracle MUST return real probabilities and Storyteller MUST return real narrative on every turn

</decisions>

<specifics>
## Specific Ideas

The retry pattern should be simple:
```typescript
async function withModelFallback<T>(
  primaryFn: () => Promise<T>,
  fallbackFn: () => Promise<T>,
  context: string
): Promise<T> {
  try {
    return await primaryFn()
  } catch (primaryError) {
    log.warn(`${context}: primary failed, trying fallback`, primaryError)
    try {
      return await fallbackFn()
    } catch (fallbackError) {
      log.error(`${context}: both primary and fallback failed`, fallbackError)
      throw fallbackError
    }
  }
}
```
