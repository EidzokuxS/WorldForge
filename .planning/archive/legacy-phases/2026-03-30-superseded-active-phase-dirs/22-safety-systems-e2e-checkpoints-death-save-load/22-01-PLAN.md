**Prerequisites:** Backend running on :3001 with an existing campaign that has world scaffold + player character (from prior E2E test phases).

**Test sequence -- 6 areas via API:**

1. **Checkpoint creation** (no LLM needed):
   - POST /api/campaigns/:id/checkpoints with { name: "E2E Test Save", description: "Before safety tests" }
   - Verify 201 response with checkpoint meta: id (string, matches timestamp-label format), name, description, createdAt (number), auto (false)
   - Verify id format matches pattern: `{timestamp}-{sanitized-name}` (e.g., "1711000000000-e2e-test-save")

2. **Checkpoint listing** (no LLM needed):
   - GET /api/campaigns/:id/checkpoints
   - Verify returns array containing the checkpoint just created
   - Verify sorted by createdAt desc (newest first)
   - Verify each entry has: id, name, description, createdAt, auto fields

3. **Checkpoint load + state consistency** (no LLM needed):
   - Record current game state: GET /api/chat/history (message count), GET /api/campaigns/:id/world (player HP, location)
   - Play 1 turn via POST /api/chat/action ("I examine the area") to change state
   - Wait for turn to complete, verify chat history grew by 2 messages
   - POST /api/campaigns/:id/checkpoints/:cpId/load to restore the saved checkpoint
   - GET /api/chat/history — verify message count matches pre-turn snapshot
   - GET /api/campaigns/:id/world — verify player HP and location match pre-turn snapshot
   - This proves checkpoint load truly reverts game state

4. **Checkpoint deletion** (no LLM needed):
   - Create a second checkpoint: POST /api/campaigns/:id/checkpoints with { name: "Temp Save" }
   - DELETE /api/campaigns/:id/checkpoints/:tempCpId
   - GET /api/campaigns/:id/checkpoints — verify temp checkpoint no longer in list
   - Verify original checkpoint still exists

5. **Auto-checkpoint via gameplay** (needs GLM):
   - This area tests indirectly — auto-checkpoint fires when HP drops to danger zone (<=2, >0)
   - Play combat-oriented turns with 60s delays, looking for auto_checkpoint SSE events
   - Turn 1: "I challenge the nearest enemy to a fight" (provoke combat)
   - Turn 2: "I attack recklessly, ignoring my defense" (invite damage)
   - Turn 3: "I continue fighting aggressively despite my wounds" (more damage)
   - Parse SSE stream for each turn — look for auto_checkpoint event type
   - Also check set_condition state_update events for HP changes (newHp field)
   - If any turn produces auto_checkpoint event: PASS (system works)
   - If HP never drops to <=2 after 3 turns: score as inconclusive (combat RNG dependent), NOT failure
   - After combat turns, restore the original checkpoint to reset state for Plan 02

6. **Death handling context** (code verification):
   - Read backend/src/engine/prompt-assembler.ts and verify SYSTEM_RULES contain HP=0 death narration rules
   - Read backend/src/engine/turn-processor.ts and verify isDowned flag propagation from set_condition
   - Verify outcome instructions (miss + weak_hit) contain isDowned=true death narration mandate
   - This is a code-level structural check since forcing HP=0 in E2E requires specific combat outcomes

**Scoring:** Each area scores pass/fail. Overall quality = (passed areas / 6) * 5.0. Minimum threshold: 4.0/5.0 (5 of 6 areas passing). Area 5 (auto-checkpoint) scores pass if event observed OR inconclusive if HP never dropped low enough.

**Rate limit handling:** Use 60s delay between GLM turns (established in Phase 20-21). Provider rate limits are NOT test failures.

**Output format:** JSON results file at e2e/22-01-results.json with per-area scores.

**After all tests:** Restore the original checkpoint via load to ensure clean state for Plan 02 browser testing.
