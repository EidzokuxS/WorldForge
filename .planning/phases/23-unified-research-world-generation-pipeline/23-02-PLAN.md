# Plan 23-02: Research Sufficiency Check Before Scaffold Steps

## Objective

Before each scaffold generation step (locations, factions, NPCs), AI evaluates whether the cached research data is sufficient for that specific step. If not, it searches for what's missing and merges the results.

## Current State

After Plan 01, ipContext is loaded from config.json. But it may not have enough detail for specific steps — e.g. the initial research might have geography facts but lack faction details.

## Tasks

### Task 1: Create research sufficiency evaluator

<read_first>
- backend/src/worldgen/ip-researcher.ts
- backend/src/worldgen/scaffold-generator.ts
</read_first>

<action>
1. In `backend/src/worldgen/ip-researcher.ts`, add a new function:
   ```typescript
   export async function evaluateResearchSufficiency(
     ipContext: IpResearchContext,
     step: "locations" | "factions" | "npcs",
     premise: string,
     seeds: WorldSeeds,
     role: ResolvedRole,
     searchConfig?: SearchConfig,
   ): Promise<IpResearchContext>
   ```

2. Implementation:
   - Use `generateObject` with a schema asking the LLM:
     - "Given these facts about {franchise}, do you have enough information to generate {step}?"
     - Schema: `{ sufficient: boolean, missingTopics: string[] }`
   - If sufficient → return ipContext unchanged
   - If not sufficient → for each missingTopic, run `webSearch()` to find more info
   - Use LLM to extract additional facts from search results
   - Merge new facts into ipContext.keyFacts (deduplicate)
   - Return updated ipContext

3. Keep it lightweight — max 2-3 additional searches per step, not a full re-research.
</action>

<acceptance_criteria>
- `ip-researcher.ts` exports `evaluateResearchSufficiency` function
- Function returns original ipContext when sufficient (no API calls)
- Function runs targeted searches when insufficient
- New facts are appended to keyFacts without duplicating existing ones
- `npm --prefix backend run typecheck` passes
</acceptance_criteria>

### Task 2: Integrate sufficiency check into scaffold generation steps

<read_first>
- backend/src/worldgen/scaffold-generator.ts
</read_first>

<action>
1. In scaffold-generator.ts, before `generateLocationsStep`, `generateFactionsStep`, and `generateNpcsStep`:
   - Call `evaluateResearchSufficiency(ipContext, step, premise, seeds, role, searchConfig)`
   - Use the (potentially enriched) ipContext for generation
   - Pass searchConfig from settings.research (provider + keys)

2. Only run sufficiency check if ipContext is not null (original worlds don't need this).

3. Report progress via SSE callback: "Checking research sufficiency for {step}..."
</action>

<acceptance_criteria>
- Each scaffold step (locations, factions, npcs) calls evaluateResearchSufficiency before generation
- Sufficiency check only runs when ipContext is non-null
- SSE progress reports sufficiency check status
- `npm --prefix backend run typecheck` passes
</acceptance_criteria>

### Task 3: Save enriched ipContext back to config

<read_first>
- backend/src/campaign/manager.ts
- backend/src/routes/worldgen.ts
</read_first>

<action>
1. After scaffold generation completes (in generate route), if ipContext was enriched:
   - Compare final ipContext.keyFacts.length with original
   - If facts were added, save updated ipContext to config.json via `saveIpContext()`
   - This ensures regeneration benefits from all accumulated knowledge

2. In regenerate-section route, same logic: if sufficiency check added facts, save them.
</action>

<acceptance_criteria>
- Enriched ipContext saved to config.json after scaffold generation
- Subsequent regenerations use the enriched research
- `npm --prefix backend run typecheck` passes
</acceptance_criteria>
