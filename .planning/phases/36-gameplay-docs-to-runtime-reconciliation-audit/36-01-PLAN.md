<artifact_contract>
The output file must:
- have one section per subsystem
- list normalized claims in a stable row-oriented format
- include source references for every claim
- explicitly note ambiguous or contradictory source language instead of resolving it silently
- include a `claim_type` field (`behavioral_rule`, `data_contract`, `ui_expectation`, `architectural_constraint`)

Granularity rules:
- one claim = one testable gameplay behavior or runtime contract
- if one sentence contains two independent behaviors joined by `and`, split it
- if one heading describes a family of sub-behaviors (for example Oracle tiers, tag categories, checkpoint fidelity), split into sub-claims rather than one umbrella statement
- target rough scale: `50-150` claims total; far below that is too coarse, far above that is likely oversplitting

Source filtering rules:
- `docs/concept.md`, `docs/mechanics.md`, and `docs/memory.md` are primary gameplay spec sources
- `docs/plans/*.md` are secondary historical sources; extract only gameplay-facing promises or design intent from them, not implementation task steps
- explicitly exclude `docs/tech_stack.md` and `docs/research.md` from the claim register unless a line in them makes a direct gameplay/runtime promise

Suggested subsystem buckets:
- turn loop and Oracle
- state mechanics
- NPC tiers and autonomy
- reflection/progression
- world engine and information flow
- memory and prompt assembly
- persistence/checkpoints
- character/start/loadout/runtime handoff
</artifact_contract>
