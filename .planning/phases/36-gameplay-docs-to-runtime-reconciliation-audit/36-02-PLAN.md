<classification_contract>
Each claim entry must include:
- claim id
- source provenance
- subsystem
- classification
- confidence
- evidence references
- short reasoning
- severity if the mismatch is gameplay-blocking or integrity-critical

Classification bar:
- `implemented_and_wired` = the full documented behavior executes in a real runtime path without manual intervention or dead seams
- `implemented_but_partial` = some live runtime path exists, but a documented sub-behavior, trigger, consequence, or data-integrity part is missing or unreliable
- `documented_but_missing` = no real runtime path satisfies the claim, even if adjacent code or dormant scaffolding exists
- `outdated_or_contradicted` = the docs claim is no longer compatible with the current architecture or later design intent

Evidence discipline:
- every `implemented_and_wired` or `implemented_but_partial` entry must cite the live code path that makes it true
- every `documented_but_missing` entry must include an explicit absence check: searched files/modules/routes and why no live runtime path exists
- confidence should be `high`, `medium`, or `low`; use `medium/low` when the code suggests behavior but true runtime proof is still inferential

Execution guidance:
- if claim count or context load becomes too large, process the matrix subsystem-by-subsystem with intermediate saves instead of doing one shallow global pass
- add an appendix for `undocumented_but_implemented` gameplay behaviors so the next milestone can decide whether to document or ignore them
- treat reflection trigger viability (`unprocessedImportance` accumulation) as a priority investigation, not an incidental note

Integrity-critical seams should be summarized separately, not buried in per-claim rows:
- rollback/retry fidelity
- checkpoint fidelity
- state authority for inventory/equipment
- reflection trigger viability
- session-scoped gameplay transport
- player-visible vs deferred post-turn simulation
</classification_contract>
</context>
