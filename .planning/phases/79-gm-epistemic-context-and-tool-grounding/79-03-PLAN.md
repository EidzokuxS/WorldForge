# 79-03 PLAN: Durable Event Discipline And Narrator Consequence Isolation

## Scope

Prevent transient local beats and failed/remote tool calls from becoming durable or player-facing truth. This plan owns `log_event` discipline, scene assembly consequence filtering, and narrator packet isolation.

Reviewer update: this plan must also cover final narration prompt assembly. A safe NarratorPacket is not enough if `assembleFinalNarrationPrompt` prepends broad world/episodic/recent-conversation sections that can contain remote/offscreen state.

## Tasks

### Task 1: RED Consequence Isolation Tests

Add tests for:

- failed tool calls do not produce player-facing `SceneEffect`;
- `spawn_npc` only creates a visible current-scene consequence when the tool result location matches the current scene/current location;
- `log_event` low-importance/transient beats are not treated as mandatory final narration facts if they are not durable.
- raw pending committed events are filtered before both `sceneEffects` and `recentContext`;
- final narration prompt does not include remote/background labels such as `Forest Outpost` when the current player scene is Shibuya.

### Task 2: GREEN Scene Assembly Filtering

Patch `summarizeToolCall` and related scene-assembly logic so it checks tool success and authoritative result location before creating player-facing effects.

If `spawn_npc` result location differs from the current scene, do not summarize it as present locally. Prefer no effect plus a diagnostic warning.

Compute filtered committed event summaries once and reuse them for both `sceneEffects` and `recentContext`. Do not append raw pending event summaries after filtering.

### Task 3: GREEN Log Event Discipline

Tighten the tool prompt contract:

- `log_event` is for durable future-relevant facts only;
- direct local beats can remain in conversation/narration without a durable event;
- examples should distinguish "paid for coffee" from "made an enemy", "found a hidden door", "promised to return", or "changed faction status";
- transient/direct service beats must not enter episodic search, location recent events, or pending committed facts unless explicitly durable.

Add either `durability: "durable"` or an enforced minimum importance rule for persisted `log_event`. This is mandatory for Phase 79; full scene-local scratch memory can remain a Phase 80/81 follow-up, but transient beats cannot continue using the durable event pipe by default.

Locked choice for Phase 79:

```ts
durability: "durable" | "scene_local"
```

Default to `scene_local` for GM player-turn plans unless the model explicitly marks the fact durable and provides a future-relevance reason. Only `durable` enters episodic events, location recent events, pending committed facts, and reflection inputs. `scene_local` can still be included in the current response/log presentation if the existing runtime has a safe local lane; otherwise it is ignored for persistence rather than smuggled into durable memory.

### Task 4: GREEN Final Narration Prompt Isolation

For the `final-visible` storyteller pass, either:

- assemble from NarratorPacket plus filtered local scene assembly only; or
- apply the model-facing redaction/filter seam to every broad `assemblePrompt` section that can contain world state, semantic memory, chronicle text, relationships, scene recent happenings, or recent conversation.

Add a regression test where global semantic/episodic memory contains `Forest Outpost` while the player is in Shibuya; the final narration prompt must not contain the remote term.

Prefer the isolated NarratorPacket path for this phase if it is smaller and clearer. If `assemblePrompt` must remain in the final-visible path, every section it contributes must be filtered through the same local model-facing view. Do not mix a safe packet with an unsafe global prefix.

Use authoritative tool results for visible summaries. For `spawn_npc`, compare `toolResult.result.locationId` to the current scene/current location id; never compare the model's requested `locationName`.
