# 79-02 PLAN: Runtime Tool Grounding And Spawn Location Safety

## Scope

Tighten runtime mutation tools so GM-selected tool plans are grounded in backend-approved local refs. This plan owns `spawn_npc` location safety and validation of free-text tool inputs that can currently bypass SceneFrame refs.

Reviewer update: this plan must introduce an execution-context seam, not only a stricter `spawn_npc` schema. Player-turn tool execution needs current scene/location and legal refs at validation time; background/offscreen execution can remain a separate explicit scope.

## Tasks

### Task 1: RED Remote Spawn Rejection

Add tests proving a player-turn `spawn_npc` cannot target a remote campaign location by free-text `locationName` when the player is in a different current scene/location.

The exact Forest Outpost regression should fail before DB mutation:

```json
{
  "name": "Outpost Cook",
  "tags": ["service-staff"],
  "locationName": "Okutama Safe Zone - Forest Outpost"
}
```

Expected:

- no NPC inserted at Forest Outpost;
- tool result is unsuccessful or GM decision validation fails;
- error text points to grounded local location refs, not "location not found".
- if a ScenePlan contains multiple tool calls and one has illegal grounding, no earlier valid mutation from that plan remains committed.

### Task 2: GREEN Grounded Spawn Contract

Update `spawn_npc` so model-facing usage prefers explicit local refs/scopes:

- `locationRef: "current_scene"` or `"current_location"` for local spawns, or `locationId` when an exact location id is exposed by the current model-facing scene view;
- optional exact location id only when exposed by the current model-facing scene view;
- `locationName` kept for compatibility but rejected if it resolves outside the current player scene/current location in player-turn execution.

Lock the schema shape unless implementation evidence forces a narrower equivalent:

```ts
locationRef: z.enum(["current_scene", "current_location"]).optional(),
locationId: z.string().optional(),
locationName: z.string().optional()
```

At least one location selector is required. New prompt examples must use `locationRef`, not `locationName`.

Keep the executor deterministic. It may resolve player current location internally, but it must not infer from prose.

`handleSpawnNpc` must return authoritative location metadata:

```ts
{ id: string; name: string; locationId: string; locationName: string }
```

Downstream scene assembly must use this result location, not the model's input string.

### Task 3: GREEN Execution Context Seam

Add an explicit grounding context to player-turn tool validation/execution:

```ts
{
  scope: "player_turn" | "actor_turn" | "background";
  subjectActorId?: string;
  currentLocationId: string;
  currentSceneScopeId: string | null;
  legalLocationRefs: Set<string>;
  legalActorRefs: Set<string>;
  legalItemRefs: Set<string>;
  legalFactionRefs: Set<string>;
}
```

The exact shape can be adjusted to local code style, but the behavior is locked:

- player-turn execution rejects location/entity refs outside the current scene/current location/exposed candidates before DB mutation;
- non-player actor turns are grounded to that actor's local context, not the whole campaign;
- background/offscreen execution must opt into a non-player-turn scope explicitly and cannot masquerade as visible player-scene mutation;
- no caller silently falls back to global name resolution for player-turn mutations.
- `executeToolCall` and ScenePlan execution must require this context for any scoped gameplay turn; no implicit default context is allowed for player turns.

Before executing any player-turn ScenePlan, run a pre-validation pass over the whole plan with this context. Either every tool input is grounded and executable, or the plan fails/repairs before the first DB mutation. Do not rely on sequential execution plus rollback as the primary safety property.

Failure contract:

- first grounding failure is sent to ScenePlanner repair once with redacted/local validation issues;
- if repair still fails, abort before mutation;
- if any failure escapes after pending committed events were queued, drain pending events for the current tick before snapshot restore/abort.

### Task 4: GREEN Tool-Input Ref Validation

Extend GM decision validation or semantic ScenePlan mapping so tools with entity-bearing inputs are checked against model-facing legal refs before execution.

Use a registry/table for every runtime tool field that resolves text or ids into entities. Cover at minimum:

- `spawn_npc.locationRef/locationId/locationName`;
- `reveal_location.connectedToName`;
- `move_to.targetLocationName`;
- `spawn_item.ownerName` when `ownerType` is `location`;
- `transfer_item.targetType: "location"` and location target fields;
- `set_relationship.entityA/entityB`;
- `add_tag/remove_tag` when target type is `location`, `item`, or `faction`;
- item names when the tool resolves them against campaign-wide inventory/entity tables.

Do not make backend decide what the player meant. Only validate that the GM-selected tool input uses a legal current/candidate ref.

Legal ref sets must include ids and display aliases only for entities intentionally exposed to the model-facing view. Name-based legacy inputs may be accepted only when they resolve to one exposed/current entity; ambiguous or remote matches are invalid.

Special cases:

- `reveal_location` may reveal a new local scene/place only through a legal parent/current anchor; it cannot use arbitrary remote `connectedToName` from global campaign data during a player turn.
- `set_relationship` is legal only for visible/current/local-known actors or explicitly exposed factions/entities.
- `transfer_item.targetType: "character"` must reject non-local/non-exposed character refs in player turns.
