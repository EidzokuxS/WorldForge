  <action>
Create an E2E test script following the established pattern from e2e/19-01-gameplay-api-tests.ts. Reuse SSE parsing utilities (parseSSEStream), campaign setup helpers, and the same CAMPAIGN_ID.

**Prerequisites:** Backend running on :3001 with an existing campaign that has world scaffold + player character + lore cards (from prior E2E test phases).

**Test sequence -- 5 areas via API:**

1. **Chat history persistence** (no LLM needed):
   - GET /api/chat/history — verify returns array of messages + premise string
   - Record message count before test
   - After gameplay turns, verify message count increased
   - Verify messages have role (user/assistant), content (non-empty string)

2. **Lore card semantic search** (needs Embedder role):
   - GET /api/campaigns/:id/lore — verify lore cards exist (from world gen pipeline)
   - GET /api/campaigns/:id/lore/search?q=relevant_term&limit=5 — verify returns results
   - Query should use a term from the campaign premise/world (e.g., "dark" or "kingdom" for Dark Fantasy campaign)
   - Verify results have: id, term, definition, category fields
   - Verify results.length > 0 (semantic search found relevant cards)
   - Test with irrelevant query (e.g., "quantum computing") — should return fewer/different results

3. **Episodic event storage via gameplay** (needs GLM):
   - Play 3 turns via POST /api/chat/action with 45s delay between turns
   - Turn 1: Exploration action ("I look around and examine my surroundings carefully")
   - Turn 2: Social action ("I approach the nearest person and introduce myself")
   - Turn 3: Significant action ("I search for hidden secrets or clues about this place")
   - For each turn: parse SSE stream, look for log_event tool calls in state_update events
   - Track event IDs from log_event results for later verification
   - Wait 10s after last turn for async embedding to complete

4. **Episodic event retrieval verification**:
   - After gameplay turns, GET /api/chat/history to verify all turns are persisted
   - Verify chat history message count increased by at least 6 (3 user + 3 assistant)
   - The episodic events were stored by log_event tool calls during turns — verify by checking that subsequent turns reference past events in narrative (evidence that prompt assembler retrieves episodic memory)

5. **Prompt assembly verification** (indirect):
   - On turns 2-3, verify the Storyteller narrative shows awareness of prior events (references things from turn 1)
   - This indirectly verifies: episodic memory retrieval, context compression, and relationship graph enrichment all feed into prompt assembly
   - Check that the narrative does NOT repeat the exact same scene description verbatim (shows context is accumulated, not reset)

**Scoring:** Each area scores pass/fail. Overall quality = (passed areas / 5) * 5.0. Minimum threshold: 4.0/5.0 (4 of 5 areas passing).

**Rate limit handling:** Use 45s delay between GLM turns (same as Phase 19). Log warnings for rate limit indicators but do NOT treat provider rate limits as test failures.

**Output format:** JSON results file at e2e/21-01-results.json with per-area scores.
