## Tasks

1. Add shared `GmToolStepResult` / step envelope: `stepId`, `attempt`, `status`, `toolName`, `candidateInput`, `validationError`, `visibleEffect`, `privateGuardTerms`, `mutationRefs`, and `settledAtTick`.
2. Add one-step candidate tool request schema using existing runtime tool schemas. Phase 81 uses existing `safeGenerateObject` contracts; provider-native tool calling is deferred.
3. Execute happy-path checklist candidate tool requests sequentially through backend validation without extra LLM calls when valid.
4. Feed backend validation errors back into one revision attempt for that step only.
5. Mark repeated illegal step as skipped/failed with reason.
6. Bounds: checklist max 6 items, total candidate tool requests max 8, one revision per step.
7. Rollback semantics: successful tool steps are settled; later failed/skipped steps do not roll back earlier successful steps. Catastrophic route/runtime failures still use existing route-level snapshot rollback.
8. Log checklist id, step id, attempt number, tool name, validation error class, status, mutation refs, settled packet correlation id, and reason without leaking private data.
9. Add tests for invalid ref -> revision success, invalid payload -> revision success, repeated invalid -> skipped, partial branch abort, and DB before/after deltas.
10. Test migration: enumerate executor/validator/semantic-scene-plan tests that survive vs. are superseded.

## Acceptance Criteria

- One bad tool step does not require repairing a giant all-actions JSON.
- Failed/skipped steps are explicit and auditable.
- Backend remains sole authority for mutations.
- Happy-path mutating turns avoid one extra LLM call per tool when checklist candidate requests validate.
