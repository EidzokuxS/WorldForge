New contract to add:
```typescript
export type WorldgenResearchUse = string;
// Known routing strings used by prompts/tests include:
// "locations", "factions", "npcs", "timeline", "power_system", "tone", "terminology".
// The schema accepts capped strings so backend does not reject unknown LLM-authored use labels as semantic errors.

export type WorldgenSourceRole =
  | "world_basis"
  | "mechanics_overlay"
  | "tone_overlay"
  | "reference_only"
  | "ambiguous";

export interface WorldgenResearchArtifactV2 {
  version: 2;
  rawPremise: string;
  rawKnownIP?: string | null;
  researchBrief: {
    interpretationSummary: string;
    ambiguityNotes: string[];
    sourceUsageRules: Array<{
      sourceLabel: string;
      role: WorldgenSourceRole;
      useFor: WorldgenResearchUse[];
      avoidFor: WorldgenResearchUse[];
      rationale: string;
    }>;
    searchJobs: Array<{
      id: string;
      sourceLabel: string;
      query: string;
      purpose: string;
      useFor: WorldgenResearchUse[];
    }>;
  };
  searchResults: Array<{
    jobId: string;
    title: string;
    description: string;
    url: string;
  }>;
  generatedContext: {
    keyFacts: string[];
    tonalNotes: string[];
    citations?: Array<{
      jobId?: string;
      url?: string;
      note: string;
    }>;
    canonicalNames?: {
      locations?: string[];
      factions?: string[];
      characters?: string[];
    };
  };
  provenance: {
    createdAt: string;
    model?: string;
    searchProvider?: string;
  };
}
```

<schema_caps>
Exact mechanical caps for `worldgenResearchArtifactSchema`:
- `rawPremise`: 2000 chars; `rawKnownIP`: 240 chars.
- `researchBrief.interpretationSummary`: 1200 chars.
- `researchBrief.ambiguityNotes`: max 8 items, each 300 chars.
- `researchBrief.sourceUsageRules`: max 8 sources.
- `sourceLabel`: 120 chars; `role`: closed `WorldgenSourceRole`; `useFor` and `avoidFor`: max 10 items each; each `WorldgenResearchUse` string max 40 chars.
- `rationale`: 500 chars.
- `researchBrief.searchJobs`: max 12 jobs; `id`: 64 chars; `query`: 240 chars; `purpose`: 500 chars.
- `searchResults`: max 48 total results; `jobId`: 64 chars; `title`: 180 chars; `description`: 700 chars; `url`: 700 chars.
- `generatedContext.keyFacts`: max 80 facts, each 450 chars.
- `generatedContext.tonalNotes`: max 30 notes, each 350 chars.
- `generatedContext.citations`: max 24 citations; `note`: 300 chars; `url`: 700 chars; `jobId`: 64 chars.
- `generatedContext.canonicalNames.locations/factions/characters`: each max 40 names, each 120 chars.
- `provenance.model` and `provenance.searchProvider`: 120 chars.

Boundary decision for `WorldgenResearchUse`: use capped strings, not a closed backend enum. This is the simpler maintainable choice because prompt routing can recognize known strings while unknown LLM-authored labels remain quoted artifact data instead of schema failures or backend semantic categorization.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user/LLM/search data -> backend artifact parser | Raw premise, LLM output, and search snippets cross into persisted backend data. |
| backend artifact formatter -> downstream prompts | Validated artifact data becomes prompt context for world generation. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-71-01-01 | Tampering | `worldgenResearchArtifactSchema` | mitigate | Validate with Zod, cap strings/arrays, and treat all premise/search fields as quoted data. |
| T-71-01-02 | Denial of Service | `parseWorldgenResearchArtifact` | mitigate | Enforce max lengths and max array sizes before downstream prompt rendering. |
| T-71-01-03 | Information Disclosure | `formatWorldgenResearchArtifactBlock` | mitigate | Render provenance metadata and bounded snippets only; do not dump hidden LLM reasoning. |
| T-71-01-04 | Tampering | source usage roles | mitigate | Preserve roles from artifact exactly; backend must not recalculate `world_basis` or overlay meaning. |
</threat_model>
