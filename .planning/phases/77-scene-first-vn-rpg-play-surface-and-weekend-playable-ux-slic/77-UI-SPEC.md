# Phase 77 - UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Phase Scope

Phase 77 turns `/game` into a scene-first solo RPG/VN play surface. The default view must answer, within five seconds:

1. Where am I?
2. Who or what is actually present?
3. What just happened?
4. What can I do next?

This phase does not redesign the whole product, add a generated-background pipeline, add a sound system, create a real QTE engine, or rewrite turn orchestration. The backend remains the source of truth for persistence, validation, concrete tool execution, state changes, inventory, locations, snapshots, and requested roll receipts. It must not be treated as the source of truth for interpreting freeform player meaning.

Sources: `77-CONTEXT.md`, `worldforge-screen-flow-contract.md`, `worldforge-visual-target-contract.md`, `worldforge-design-lab-results.md`, `marinara-gm-flow-reference.md`, `REQUIREMENTS.md` P77-R1 through P77-R8.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline padding, beat metadata gaps |
| sm | 8px | Compact element spacing, chip gaps, drawer row gaps |
| md | 16px | Default element spacing, dock internal padding |
| lg | 24px | Scene HUD padding, drawer padding, grouped panel padding |
| xl | 32px | Scene-to-dock breathing room, desktop rail gap |
| 2xl | 48px | Major layout divisions, desktop shell margins |
| 3xl | 64px | Wide desktop scene breathing space only |

Exceptions:

- Icon-only rail and HUD controls must be at least 44px by 44px on touch surfaces.
- Mobile bottom dock must respect `env(safe-area-inset-bottom)` and keep at least 16px usable padding.
- Scene viewport must keep stable aspect framing: desktop minimum visual scene height 55vh; mobile minimum visual scene height 36vh before the dock.
- Narration text measure must cap at 720px on desktop and use full available width minus 32px on mobile.

## Typography

Use exactly these four type sizes and two weights for the Phase 77 live play surface:

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Label | 12px | 600 | 1.2 |
| Body | 16px | 400 | 1.5 |
| Heading | 20px | 600 | 1.2 |
| Display | 28px | 600 | 1.15 |

Rules:

- Allowed weights: regular 400 and semibold 600 only.
- UI chrome uses Inter.
- Narration in `NarrationDock` may use Playfair Display at Body size only; it must stay inside the game text box and must not become a long newspaper column.
- Do not scale font size with viewport width in the live play shell. Use responsive layout, not viewport-scaled type.
- Uppercase micro-labels are allowed for HUD/drawer section labels, but letter spacing must stay at or below `0.12em`.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#09090B` / `--background` | Live play page background, scene shadows, dark cinematic negative space |
| Secondary (30%) | `#18181B` and `#27272A` / `--card`, `--secondary` | Narration dock, drawers, HUD, widget rail, compact panels |
| Accent (10%) | `#E63E00` / `--color-blood` | `Continue`, Send, current beat marker, active drawer tab, high-salience dice/choice cue |
| Destructive | `#EF4444` | Delete checkpoint, irreversible load/restore warnings, destructive confirmation only |

Accent reserved for: `Continue`, Send, current beat/progress marker, active widget/drawer state, dice/choice beat emphasis, actor focus pulse when an actor enters or speaks.

Restrictions:

- Do not use the accent for all buttons, static borders, decorative gradients, or generic hover states.
- Inspect/debug can use `--color-mystic` (`#3B82F6`) as a low-opacity signal color, but it must not compete with primary play actions.
- Rain/neon/weather tints are allowed as low-opacity scene overlays only. They are environmental presentation, not a second interaction accent.
- No cream, parchment, tan, newspaper, editorial-reader, or warm paper dominant surface on `/game`.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Continue |
| Empty state heading | Scene loading |
| Empty state body | WorldForge is grounding the opening scene. You can act once the first beat appears. |
| Error state | The turn could not finish. Retry the action, continue from the last settled beat, or open Inspect for details. |
| Destructive confirmation | Delete checkpoint: Delete this checkpoint permanently? |

Additional required copy:

| Element | Copy |
|---------|------|
| Send tooltip | Send action |
| Continue tooltip | Let the scene breathe without adding a specific action |
| Next tooltip | Show next beat |
| Auto toggle label | Auto |
| Log button label | Log |
| Inspect drawer empty | No mechanics for the current beat. Raw details appear here only when available. |
| World drawer empty | No discovered world details for this scene yet. |
| Inventory empty | You are not carrying anything visible right now. |
| Journal empty | No clues, notes, or records have been pinned yet. |
| Character drawer empty | Select a visible actor or open your character sheet. |
| Load checkpoint confirmation | Loading replaces the current timeline with this checkpoint. Continue? |
| Leave game confirmation | Progress is saved automatically. Return to the campaign library? |

Copy rules:

- Player-facing labels must say `Log`, `World`, `Inventory`, `Journal`, `Character`, `Inspect`, `Saves`; avoid implementation labels such as `Support Actions`, `scene-settling`, raw event names, or `authoritative turn completion`.
- Mechanic beats must be fiction-facing: `Clean success`, `Costly success`, `Miss`, `Bad break`, `Close call`. Raw chance, roll, reasoning, JSON, and payloads belong in Inspect.
- The live input is one raw narrative text field plus visible `Continue`. Do not require `Act`, `Speak`, `Observe`, or similar action-category modes.

## Live Play Visual Contract

### Default Layout

`/game` default layout must be:

1. `GameSceneShell` as the full viewport root.
2. `SceneBackdrop` as the first visual read.
3. `SceneHUD` as a thin overlay, not a page toolbar.
4. `PresenceLayer` over the scene.
5. `StageOverlay` for persistent current-beat signals/effects.
6. `NarrationDock` and `ActionDock` anchored to the lower scene.
7. `WidgetRail` as compact game tools.
8. `DrawerHost` for Log, World, Inventory, Journal, Character, Inspect, and Saves.

Current permanent side panels must not remain as desktop columns. `LocationPanel`, `CharacterPanel`, `LorePanel`, `OraclePanel`, `NarrativeLog`, `QuickActions`, `ActionBar`, and `CheckpointPanel` may be reused as data/rendering sources, but their default homes change:

| Current surface | Phase 77 home |
|-----------------|---------------|
| `LocationPanel` | `WorldDrawer` plus scene/HUD summary |
| `CharacterPanel` | `CharacterDrawer` plus PC HUD chip |
| `LorePanel` | `JournalDrawer` and Inspect/Lore tab |
| `OraclePanel` | `InspectDrawer`; fiction-facing mechanic beat in `NarrationDock` |
| `NarrativeLog` | `LogDrawer`; latest assistant turn becomes `DisplayBeat[]` in `NarrationDock` |
| `QuickActions` | Choice/action chips after beat handoff, not `Support Actions` copy |
| `ActionBar` | `ActionDock` with one raw narrative input, Send, and first-class Continue |
| `CheckpointPanel` | `Saves` drawer/modal opened from `WidgetRail` or HUD |

### Rejected Visual Direction

Do not implement:

- cream document backgrounds,
- parchment/newspaper/editorial-reader panels,
- long prose columns as the default play surface,
- permanent debug/admin sidebars,
- Marinara's exact chrome, spacing, widgets, or visual skin,
- generic purple sci-fi dashboard styling,
- `backdrop-blur` as a substitute for actual layout/readability structure,
- nested cards inside cards.

Marinara is a flow reference only: scene background, bottom narration box, staged beats, Next/Auto/Log, game widgets, actor chips, drawers, choices/dice/text effects. WorldForge must preserve stronger backend authority and use its own dark cinematic shell.

## Component Contract

### GameSceneShell

Purpose: full-screen scene-first container for `/game`.

Visual:

- Occupies `h-dvh` with dark cinematic background.
- No root page cards. The scene is the page.
- Desktop: scene occupies the full viewport with lower dock overlay; no permanent left/right admin columns.
- Mobile: vertical scene -> narration -> action stack; widget rail becomes bottom or edge icon row.

Interaction:

- Owns drawer state, beat playback state, and per-campaign draft restoration.
- Must preserve current draft when any drawer opens/closes.
- Must expose data test hooks for visual gates: `game-scene-shell`, `scene-backdrop`, `narration-dock`, `action-dock`, `widget-rail`, `drawer-host`.

### SceneBackdrop

Purpose: establish the current place before text is read.

Visual:

- Full-bleed or viewport-dominant visual layer.
- Acceptable first slice: CSS/stylized scene built from location mood, time, weather, tags, and a generated/placeholder image when available.
- Must include concrete place cues: scene title, broad location, weather/time/mood tint, at least one visible environmental detail.
- Must not be blank, cream, or purely abstract gradient.

Interaction:

- Presentation only. It must not mutate world state.
- Can fade/tint/flash on display events after backend state settles.

### StageOverlay

Purpose: persistent visual stage signals for the current beat.

Visual:

- Renders compact, scene-attached overlays such as whispers, side remarks, glitch text, danger lines, flash/fade/screen-shake states, or speaker-adjacent callouts.
- Signals remain visible while their display beat is active and clear only when `Next` advances past that beat or a new backend turn boundary replaces the beat list.
- Stage signals must not auto-disappear on timers that outrun the player.
- Wide desktop space should be used by scene composition, actor markers, signals, or compact scene widgets rather than left/right empty bands.

Interaction:

- Presentation only. Stage signals cannot commit state, select hidden actors, or call backend routes.
- Clicking a signal may open `Log` or `Inspect` only if the signal explicitly represents existing inspectable data.

### SceneHUD

Purpose: compact game status without admin weight.

Visual:

- Thin top overlay.
- Shows current scene name, broad location, time/weather/mood if known, turn status, and compact Home/Saves/Settings controls.
- Uses icon buttons for Home, Saves, Settings, volume/image/debug if present. Provide accessible labels/tooltips.

Interaction:

- Turn status copy must be player-facing: `Reading`, `Auto`, `Thinking`, `Settling`, `Ready`.
- `scene-settling`, raw event labels, provider names, and debug toggles must not appear unless Inspect/debug mode is open.

### PresenceLayer

Purpose: show who/what is actually present.

Visual:

- Visible/interactable actors appear as foreground chips, silhouettes, portraits, or markers.
- Sensed/same-area nearby actors appear as muted signals, not as interactable portrait chips.
- Off-screen anchors do not appear in the scene layer; they belong in `WorldDrawer` or `JournalDrawer`.

Interaction:

- Clicking a visible actor opens `CharacterDrawer` scoped to that actor.
- Non-visible actors cannot be selected as direct scene targets from the scene layer.
- Actor states required: `visible/interactable`, `sensed nearby`, `off-screen anchor`.

### NarrationDock

Purpose: staged VN/RPG latest-beat reader.

Visual:

- Bottom or lower-center game text box over the scene.
- Shows only the latest turn's current beat by default, not full scrollback.
- Displays speaker/name treatment for dialogue where available.
- Contains `Next`, `Auto`, and `Log` controls.

Interaction:

- `Next` advances local display beat only.
- `Auto` advances local display beats on a readable timer and can be paused.
- `Log` opens `LogDrawer`.
- None of `Next`, `Auto`, or `Log` may call backend turn routes.
- Choices, dice/mechanic summaries, and temporary pressure beats appear above or inside the dock as game beats.

### DisplayBeat Adapter

Purpose: local presentation model over existing messages/SSE events.

Required beat kinds:

| Kind | Default presentation |
|------|----------------------|
| `narration` | prose beat in `NarrationDock` |
| `dialogue` | speaker callout plus line |
| `side_remark` | compact actor reaction |
| `mechanical_result` | fiction-facing result chip, raw details hidden |
| `state_change` | compact consequence beat |
| `choice` | choice/action cards after control returns |
| `progress` | `Thinking`/`Settling` status, no raw event names |
| `input_handoff` | unlocks `ActionDock` and quick choices |

Rules:

- DisplayBeat is presentation state, not authority.
- It may adapt from existing `ChatMessage`, `debugReasoning`, quick actions, Oracle result, travel feedback, and world data.
- It must not invent committed outcomes. If data is missing, show a neutral display state rather than fake mechanics.
- Visible chunk/beat counters such as `chunk 1/3`, `beat 2/5`, or debug progress are forbidden on the player-facing dock.

### ActionDock

Purpose: freeform player action attached to the current scene.

Visual:

- Anchored below/inside `NarrationDock`.
- Contains multiline raw narrative input, Send icon button, and `Continue` button.
- Does not contain required `Act`, `Speak`, `Observe`, or equivalent action-category controls.
- Compact chips may appear only after narration hands control back.

Interaction:

- `Send` creates a backend turn with the input text.
- `Continue` creates a backend turn with an explicit continue action (`Continue scene.` payload is route compatibility, not visible UI copy).
- `Continue` uses the exported `CONTINUE_ACTION_PAYLOAD`; do not duplicate the literal payload inline in page components.
- Empty input is valid only through `Continue`, not Send.
- Draft persists per campaign in frontend local/session storage and survives drawer navigation.
- Enter submits only when this matches existing keyboard behavior and the user can still insert new lines with Shift+Enter.

### WidgetRail

Purpose: compact game tools, not permanent data columns.

Required widgets:

| Widget | Opens |
|--------|-------|
| Log | `LogDrawer` |
| World/Map | `WorldDrawer` |
| Inventory | `InventoryDrawer` |
| Journal | `JournalDrawer` |
| Character | `CharacterDrawer` |
| Inspect | `InspectDrawer` |
| Saves | Save/checkpoint drawer or modal |

Rules:

- Use lucide icons with accessible labels/tooltips.
- Desktop rail may sit right or left over the scene.
- Mobile rail becomes a bottom row or compact floating menu.
- Active drawer state can use accent. Inactive widgets stay neutral.

### DrawerHost

Purpose: one overlay system for secondary information.

Visual:

- Drawer overlays dim the scene, slide over it, and keep the scene visibly behind.
- Desktop drawer width: 420px to 560px.
- Mobile drawer: full-width bottom sheet or full-height panel.
- Do not place drawer panels inside other cards.

Interaction:

- Escape/click outside closes drawer unless a destructive confirmation is open.
- Drawer open/close must not clear input draft, beat index, active scene, or selected address mode.
- Drawer content can scroll independently from the scene.

### InspectDrawer

Purpose: optional audit path for mechanics and raw system data.

Tabs:

- `Beat`: fiction-facing mechanic summary for the current beat.
- `Oracle`: chance band, roll, outcome, short reason.
- `State`: PC status, constraints, current scope.
- `Events`: canonical turn timeline.
- `Debug`: raw payloads/logs/reasoning, visible only when debug mode is enabled.

Rules:

- Closed by default.
- Raw Oracle math, raw reasoning, JSON, payloads, and debug event names appear only here.
- The player must never need Inspect to understand the scene consequence.

### WorldDrawer

Purpose: known world, movement, and scene scope.

Content:

- Current macro location, persistent scene location, and immediate scene.
- Known paths/travel options.
- Local history/recent happenings.
- Presence groups: visible now, sensed nearby, same broad area, off-screen anchors.
- Known factions/pressure if available.

Rules:

- Clearly distinguish `visible/interactable now` from `same broad area`.
- Travel actions call backend only from explicit player action/choice controls.
- World data must be human-readable, not raw scaffold fields.

### LogDrawer

Purpose: full history without making the default play surface a scrollback log.

Content:

- Full session message history.
- Retry, undo, edit controls remain available where currently supported and safe.
- Filters/tabs may be minimal: `Story`, `Player`, `Mechanics`.

Rules:

- Opening Log must preserve current `NarrationDock` beat index and input draft.
- Latest turn stays readable in `NarrationDock`; Log is secondary.

### InventoryDrawer

Purpose: items as game objects.

Content:

- Equipped items, carried items, scene items if available.
- Item tags and signature/equipped state.

Rules:

- Empty state uses the copywriting contract.
- Do not expose raw item JSON.
- Item actions can be disabled if mechanics are not in scope; disabled state must explain why.

### JournalDrawer

Purpose: lore, clues, records, and notes.

Content:

- Lore cards grouped by concept/rule/location/faction/NPC/ability/item/event.
- Clues/notes/session records if available.
- Search field may reuse current lore search behavior.

Rules:

- Label as `Journal`, not `World Lore` in the rail.
- Search errors keep existing content and show a compact error state.
- Raw provenance/debug stays in Inspect, not Journal default.

### CharacterDrawer

Purpose: PC sheet and actor detail.

Content:

- PC summary from current `CharacterPanel`.
- Visible actor detail when opened from `PresenceLayer`.
- HP/vitality, appearance, tags, equipment, carried items.
- Advanced/raw record behind Inspect/debug only.

Rules:

- Default view is human sheet first.
- It must not duplicate every raw character field from review UI.
- If opened from an actor chip, the actor name and visible scene relation appear at top.

## Interaction Rules

| Control | Backend turn? | Required behavior |
|---------|---------------|-------------------|
| Next | No | Advances local beat index only |
| Auto | No | Toggles local beat timer only |
| Log | No | Opens `LogDrawer` only |
| Send | Yes | Sends current action text as player turn |
| Continue | Yes | Sends explicit continue/let-scene-breathe action |
| Drawer open/close | No | Preserves draft, beat index, address mode, scene |
| Choice chip | Yes, if selected as player action | Sends the selected action text after player click |
| Inspect tabs | No | Reads details only |
| Travel path | Yes, if selected as action | Sends movement action through existing turn path |

Processing states:

- While streaming/thinking/settling, disable duplicate `Send` and `Continue`.
- Keep `Next`, `Auto` pause, `Log`, and Inspect available when technically safe.
- Show compact progress in `SceneHUD` and `NarrationDock`; do not show raw SSE event labels on the default surface.

## Desktop Acceptance Gates

Run at modern desktop-first viewports `2560x1440`, `1920x1080`, and `1728x1117`, with `1440x900` and `1366x768` as lower-bound regression widths.

| Gate | Must pass |
|------|-----------|
| Five-second game read | Screenshot reads as a VN/RPG scene, not an article, document, dashboard, or debug cockpit |
| Scene-first hierarchy | Backdrop/location layer is visible before prose; `NarrationDock` is lower scene overlay |
| No permanent admin columns | Location, Character, Lore, Oracle, and raw log are not permanent sidebars |

| Debug containment | Default screenshot does not show `Oracle`, `Chance`, `Roll`, `Raw reasoning`, `JSON`, or raw event labels |
| Cadence controls | `Next`, `Auto`, and `Log` visible in `NarrationDock`; Next/Auto/Log do not call backend turn routes |
| Backend turn controls | `Send` and `Continue` call backend turn routes |
| Drawer persistence | Type a draft, open/close Log, World, Inspect, Character, Inventory, Journal; draft remains |
| Dice/choice beat | At least one mechanic/choice beat appears as game UI, not raw debug |
| Text fit | Labels and buttons fit without overlap at both desktop widths |

| Scene/dock stack | Scene, current beat, and input stack vertically without permanent sidebars |
| Touch targets | Rail/HUD/action controls are at least 44px by 44px |
| Drawer behavior | Drawers become full-width bottom sheets or panels and preserve input draft |
| No text overlap | HUD, actor chips, beat text, controls, and input do not overlap |
| Debug hidden | Raw Oracle/debug/JSON is hidden until Inspect is explicitly opened |
| Continue access | `Continue` remains visible without requiring text input |
| Keyboard resilience | Mobile keyboard does not hide the active input or Send/Continue controls |

## Visual QA Checks

Required before implementation closeout:

1. Capture desktop and mobile screenshots of default `/game`.
2. Reject if more than one-third of the first viewport reads as white/cream/parchment/tan paper.
3. Reject if the screenshot's dominant shape is a long prose column.
4. Reject if default `/game` shows permanent Location/Character/Lore/Oracle sidebars.
5. Reject if raw debug terms are visible by default: `Raw reasoning`, `JSON`, `Chance`, `Roll`, `scene-settling`, `payload`, `support actions`.
6. Verify `Next`, `Auto`, and `Log` are local by spying on network/route calls.
7. Verify `Send`, `Continue`, travel action, and choice chip use the backend turn route.
8. Verify drawer open/close does not mutate the input draft.
9. Verify visible actor chip opens `CharacterDrawer`; same-area/off-screen actor does not appear as a direct scene chip.
10. Run a 10-turn live or deterministic playtest: Continue, freeform action, actor interaction, movement/world inspection, one drawer open, and one dice/choice beat.
