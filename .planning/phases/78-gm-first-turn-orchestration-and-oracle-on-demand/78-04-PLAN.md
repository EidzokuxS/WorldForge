## Tasks

### Task 1: RED Orchestration-Order Assertions

**RED**

- Ensure tests assert the order:
  - neutral frame built before GM decision;
  - GM decision called before Oracle;
  - Oracle called only for a GM `roll_oracle` decision;
  - tool/combat paths never request a new roll directly. If uncertainty is needed before a tool/combat outcome, the GM must first return `roll_oracle`; a later tool/combat continuation may only consume the existing roll result and must not make another Oracle call from the tool/combat path.
  - ScenePlanner receives the GM decision and optional Oracle result;
  - validation/execution/final narration still run after planning.
- Add route-stream tests proving no `oracle_result` SSE event is emitted for direct speech, observation, clarification, or Continue.

**GREEN**

- Modify `processTurnScenePlan` to call `runGmTurnDecision` after `buildSceneFrame`.
- Remove unconditional `callOracle`.
- Build an Oracle payload only from `roll_oracle.rollRequest` evidence. Backend may still compute deterministic tags/candidate context from concrete GM refs, but must not parse raw text to invent them.
- Emit `oracle_result` only when a backend roll actually occurs.
- Pass `oracleResult: null` or omit it for direct/continue/clarification paths, and preserve final narration/event tail.
- Normalize route compatibility at the orchestration boundary: `playerAction` is the only semantic raw input; `intent` must be treated as a mirror and `method` as empty compatibility data. Do not forward `intent`/`method` into Oracle, target, combat, or ScenePlanner prompts as separate product meaning.
- Remove or rename `intent`/`method` from `turn.begin` structured log payload so observability does not imply they are authoritative semantics; prefer a single `rawInput` or compatibility note.
- Construct the direct/continue/clarification zero-action/no-mutation artifact in this layer, then feed it through validation/execution/finalization instead of bypassing the backend rulebook tail.
- Clarification path contract: return the GM `clarificationPrompt` through the existing turn stream as one existing `narrative` SSE event, optionally preceded by existing `scene-settling` progress, followed by the normal `finalizing_turn` and `done` tail. This is the same event family already parsed by `frontend/lib/api.ts` `parseTurnSSE` and rendered by `/game` through `onNarrative`; do not add a new event type or toast-only path. It must not emit `oracle_result`, `state_update`, or `quick_actions`, must not execute tools, and must not advance persisted world state beyond existing rollback-safe turn bookkeeping.

### Task 2: Preserve Rollback And Finalization Boundaries

**RED**

- Add tests where GM decision succeeds but ScenePlanner/validation/execution fails; the route must restore the pre-turn snapshot.
- Add a test where requested Oracle fails; the route must restore the snapshot and clear the last-turn snapshot state.

**GREEN**

- Keep `/action` snapshot capture before turn processing and restoration on any generator failure.
- Ensure GM decision, optional Oracle, ScenePlanner, validation, execution, storyteller narration, and rollback-critical post-turn work remain inside the same failure boundary.
- Preserve existing `finalizing_turn`, `state_update`, `quick_actions`, and `done` event semantics. Clarification emits existing `scene-settling` only if needed for progress, then `narrative` + `finalizing_turn` + `done` only.

## Acceptance Criteria

- Pure speech, obvious observation, guaranteed actions, and Continue do not call Oracle and do not emit `oracle_result`.
- Requested uncertainty calls Oracle once and records a backend roll receipt.
- Backend no longer uses raw player text to decide target/hostility/combat before GM decision.
- Rollback, snapshots, validation/execution, storyteller finalization, and SSE shape remain intact.

## Rollback Notes

If orchestration changes break rollback or finalization, revert this plan's `turn-processor.ts` and route edits while keeping GM decision and neutral frame modules. Do not restore unconditional Oracle as a shortcut; repair the failure boundary.
