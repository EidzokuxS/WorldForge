## Tasks

### Task 1: RED Concrete Ref And Tool Validation

**RED**

- Add tests proving:
  - invented actor IDs/labels fail mapping;
  - background/forbidden/offscreen actors cannot be direct targets;
  - unsupported tools fail even if GM asks for them;
  - missing target/tool semantics produce clarification/failure, not backend invention;
  - tool inputs cannot mutate time/stats/inventory/location except through legal runtime tools.

**GREEN**

- Tighten semantic mapping and validation so GM outputs must reference frame candidates or allowed tools.
- Preserve backend ID generation and existing `runtimeToolInputSchemas`.
- Do not add fallback target selection from raw text.
- Do not accept LLM-authored direct state deltas outside validated tool calls.
- Validate the zero-action/no-mutation artifact used by direct/continue/clarification paths. It may carry narration guidance/event metadata, but must not mutate time, stats, inventory, location, relationships, HP, or conditions except through legal runtime tools.

**VERIFY**

```bash

```

### Task 2: Combat Envelope After GM Concrete Choice

**RED**

- Add tests proving `"I hit Iru"` does not trigger combat math unless GM decision selects a concrete target/combat path.
- Add tests proving combat envelope/bounds are derived when GM chooses combat transition with a valid actor/target candidate.
- Add tests proving invalid/offscreen combat target fails closed or returns clarification without executing combat tools.

**GREEN**

- Adapt combat envelope construction to consume GM decision or validated plan context with concrete actor/target refs.
- Keep qualitative deterministic math from `buildCombatEnvelope` and narrative bounds intact.
- Keep `isHostileCombatAction` out of pre-GM player-turn orchestration; if retained, it must not be used as product truth for raw player text before GM/Judge.
- Add a code comment or static assertion around `isHostileCombatAction` stating it is forbidden from player-turn pre-GM orchestration and allowed only for NPC/internal or explicitly GM-selected combat helper paths.
- Add regression coverage proving NPC tick/agent/internal consumers still derive combat/oracle context when their own concrete framing requires it.

## Acceptance Criteria

- Backend validates all GM-selected refs/tools against candidates, allowed tools, schemas, and world rules.
- Backend never invents missing target/tool semantics to make validation pass.
- Combat math runs only after GM/Judge concrete combat selection.
- Illegal LLM-authored state changes cannot overwrite backend world truth.

## Rollback Notes

If tightened validation blocks legitimate existing tools, revert only the narrow validator change and add a failing test that captures the legitimate tool contract. Do not loosen validation by accepting invented refs or raw-text target fallback.
