# 78-02 PLAN: Neutral Scene Packet Boundary

## Scope

Make `SceneFrame` a neutral evidence and affordance packet for player turns. It must provide state, candidate IDs/names, visibility/awareness bands, recent events, movement candidates, memory hints where already supported, and allowed tools without deriving target, hostility, combat mode, Oracle context, or action category from raw player prose.

Planning note: "memory hints" here means only already-existing recent-event/deferred-hook/context surfaces that the current code exposes. Phase 78 must not invent a new memory-hint field or make one mandatory unless source impact analysis discovers an existing executable contract to preserve.

## Tasks

### Task 1: RED Neutral Frame Assertions

**RED**

- Ensure the tests from 78-01 assert:
  - target candidates are listed with IDs/names/visibility bands;
  - broad-location/background actors are not direct interaction targets;
  - `oracleContext` is `null`/absent before GM decision;
  - `combatEnvelope` is `null`/absent before GM decision even when raw text says `"attack"`, `"hit"`, or another hostile word.

**GREEN**

- Change `buildSceneFrame` so player-turn frame construction does not call raw-text `resolveOracleContext` or raw-text `deriveCombatEnvelope`.
- Preserve existing candidate, roster, perception, recent event, movement candidate, deferred hook, and `allowedTools` construction.
- If existing consumers still need explicit GM-provided target/combat derivation, expose a separate helper that accepts concrete GM-selected refs/hostile framing. Do not call that helper from neutral frame construction.
- Do not remove deterministic combat math helpers; only remove their pre-GM raw-text authority from player-turn frame creation.
- Treat `frame.playerAction` as raw text for the GM seam only. No non-GM prompt builder may consume `playerAction`, `intent`, or `method` as already-interpreted target, hostility, action category, or Oracle evidence.
- Strip `oracleContext` and `combatEnvelope` keys from any pre-GM prompt serialization if their value is `null`/`undefined`, so their mere shape does not bias downstream models.

**VERIFY**

```bash

```

### Task 2: Preserve Rulebook Facts And Affordances

**RED**

- Add or keep tests proving neutral frame still carries:
  - active/support/background roster bands;
  - forbidden/background actor references for validation;
  - movement candidates with IDs/names;
  - allowed runtime tool names from backend schemas.

**GREEN**

- Keep `SceneFrame` type compatible enough for existing validation/execution paths, but make semantic fields explicitly optional/nullable and documented as post-GM only.
- Ensure no missing field causes `semanticScenePlanToStrictPlan` or `validateScenePlan` to fabricate defaults that imply target/combat meaning.
- If there is no concrete current memory-hint field, add a test or code comment proving the neutral packet still preserves existing recent events/deferred hook context and does not invent a new memory surface.
- Add a regression proving NPC/internal paths that legitimately need Oracle/combat context still receive it through their own concrete framing. Player-turn neutralization must not break NPC tick/agent consumers.

## Acceptance Criteria

- Neutral `SceneFrame` never authoritatively infers target, hostility, combat, action category, or Oracle context from `playerAction`, `intent`, or `method`.
- All backend-owned state/candidate/visibility/tool facts remain present for GM/Judge and validation.
- Existing ScenePlan mapping can still reject forbidden/offscreen refs without broad-location target leakage.

## Rollback Notes

If this plan regresses candidate availability, roll back only the SceneFrame changes and preserve the RED tests. Do not restore raw-text target or hostility inference; repair candidate collection instead.
