# 80-01 PLAN: Forecast Contract And Advisory Storage

## Scope

Define the forecast contract as bounded advisory GM memory, plus the minimal storage/staging seam needed for later turn integration. This plan must not make forecasts executable or player-visible by default.

Review amendment: this plan must also resolve the persistence boundary before source execution. Forecast storage is staged in memory during a turn and committed only after final narration succeeds. Phase 80 must not store forecasts in vectors or arbitrary sidecars; if campaign config is used, tests must prove unknown config fields are preserved and snapshot/restore covers the write.

## Tasks

### Task 1: RED Forecast Schema Tests

Add `world-forecast.test.ts` proving:

- forecast entries are capped by count/string length;
- forecast entries require horizon/base tick/subject refs/confidence/privacy/player-facing eligibility;
- forecast schemas reject direct deltas such as `hpDelta`, `locationId`, `inventoryAdd`, `relationshipDelta`, `durableEvent`, `plannedActions`, and `toolInput`;
- forecast schemas reject nested executable smuggling such as `metadata.toolInput`, `revealBudget.durableEvent`, `candidateRefs.locationId`, and arbitrary payload-like objects;
- a forecast can serialize and restore without gaining executable fields.

### Task 2: GREEN Forecast Contract

Implement `world-forecast.ts` with:

- strict Zod schemas for `WorldTrajectoryForecast`, `ForecastEntry`, and staged forecast updates;
- helper constructors/parsers for safe load/restore;
- bounded default caps;
- advisory-only metadata names that do not look like runtime tool payloads.

### Task 3: Storage/Staging Decision

Implement the smallest storage seam that preserves rollback safety:

- prefer staged forecast updates until turn success;
- commit staged forecast only after final narration succeeds and route-level snapshot restore will not run;
- only persist if persistence is covered by current route snapshot behavior or an explicit post-success write;
- if persistence uses campaign config, prove unknown config fields are preserved and restored;
- do not persist forecasts to vector stores or non-snapshot sidecars in Phase 80;
- document in code/tests why forecast storage is advisory and not world truth.

## Acceptance Criteria

- Forecast schema exists and is strict.
- Forecast cannot carry executable state deltas.
- Forecast updates can be staged without changing DB/world truth.
- Persistence boundary is concrete and rollback-safe.
- Tests prove advisory-only contract before any prompt integration.
