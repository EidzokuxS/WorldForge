# 80-02 PLAN: Forecast Refresh, Scope Metadata, And Invalidation

## Scope

Build or refresh forecasts from durable world facts and invalidate stale forecast entries after relevant committed changes. Scene-local transient events must not become long-horizon forecast pressure.

Review amendment: this plan owns the actual LLM-authored forecast builder, not only refresh helpers. Backend assembles deterministic durable facts; the GM/LLM writes bounded advisory pressure; backend validates and stages the result. Backend must not infer semantic forecast content from raw prose.

## Tasks

### Task 1: RED Invalidation Tests

Extend `world-forecast.test.ts` proving:

- durable committed events invalidate relevant forecast entries;
- movement/spawn/relationship/condition-like committed effects invalidate entries scoped to those refs;
- scene-local `log_event` does not create or invalidate long-horizon forecast pressure;
- failed turns do not persist newly refreshed forecasts.

### Task 2: GREEN Forecast Refresh/Inval Helpers

Add helpers that:

- decide whether a forecast is missing, expired, invalidated, or still usable;
- derive refresh inputs only from durable, backend-known facts;
- produce invalidation metadata keyed by tick/event/tool result refs;
- avoid semantic backend interpretation of raw player prose.

### Task 3: GREEN LLM Forecast Builder

Implement a `runWorldForecastBuilder`-style function that:

- receives a bounded durable-fact packet, current tick, campaign id, and prior usable forecast;
- uses structured output with the strict forecast schema from 80-01;
- asks the model for advisory pressures, likely next beats if nothing changes, and preconditions;
- forbids state deltas, runtime refs outside backend-known facts, tool payloads, and player-facing omniscience;
- returns a staged forecast revision and concise diagnostics, not persisted world truth.

### Task 4: Rollback-Safe Persistence Hook

Wire staging/persistence so a forecast refresh can be prepared during turn processing but committed only after turn success. If implementation needs route-level storage, keep the write inside existing snapshot/restore semantics.

## Acceptance Criteria

- Forecast refresh and invalidation use durable facts, not raw prose or scene-local transient text.
- Forecast content is authored by GM/LLM from durable facts, not invented by backend heuristics.
- Forecast refresh is lazy: missing, expired, invalidated, or major committed durable change.
- Failed turns do not persist forecast revisions.
- Invalidation behavior is tested before prompt integration.
