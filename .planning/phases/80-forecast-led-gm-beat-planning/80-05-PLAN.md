# 80-05 PLAN: Turn Processor And ScenePlan Integration

## Scope

Insert forecast scoping and BeatPlan generation into the normal `/action` turn path before ScenePlan/tool execution and before final narration. ScenePlan must use BeatPlan as the current-beat intent contract, while backend validation/execution remains authoritative.

Review amendment: BeatPlan is a hard gate. If BeatPlan is missing, invalid, or throws, the turn must not reach ScenePlanner, tool execution, final narration prompt assembly, narration model call, chat append, or forecast commit.

## Tasks

### Task 1: RED Turn Order Tests

Extend turn-processor tests proving:

- normal `/action` turn order is SceneFrame -> scoped forecast -> GM decision -> BeatPlan -> optional Oracle -> ScenePlan -> validation -> execution -> narrator packet;
- invalid/missing/throwing BeatPlan prevents `runScenePlanner`, `executeScenePlan`, `assembleFinalNarrationPrompt`, narration model call, `appendChatMessages`, and forecast commit;
- direct/no-roll, continue, and clarification paths still receive a non-mutating BeatPlan before final narration, unless clarification is explicitly documented as a no-narration exception and tested;
- failed post-BeatPlan turns do not commit staged forecast updates.

### Task 2: GREEN Turn Integration

Wire the turn path:

- load/stage/refresh forecast as needed;
- scope forecast for the local frame;
- pass scoped forecast to GM decision every turn, even when empty;
- call `runGmBeatPlanner` after GM decision and before ScenePlan;
- pass BeatPlan to ScenePlanner formatting;
- keep Oracle optional and GM-selected;
- update `SCENE_PLAN_TURN_ORDER` so tests pin the new order.

### Task 3: ScenePlanner Grounding Guard

Make ScenePlanner aware that forecast refs are advisory only:

- forecast refs cannot become legal actor/location/item refs unless already present in Phase 79 tool execution context;
- forecast and BeatPlan refs do not add candidates and do not expand `ToolExecutionContext`;
- ScenePlan validation rejects coarse posture/tool-category mismatches against BeatPlan before mutation;
- illegal forecast-only refs fail validation before mutation;
- diagnostics use reason codes, not hidden/private name dumps.

## Acceptance Criteria

- No normal turn reaches tools or final narration without BeatPlan.
- Forecast-only refs cannot be executed.
- ScenePlan cannot introduce unrelated tool categories after BeatPlan narrows or forbids mutation.
- Backend validation/execution still owns all state changes.
